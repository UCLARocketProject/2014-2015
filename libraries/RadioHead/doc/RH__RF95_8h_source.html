212 /// \class RH_RF95 RH_RF95.h <RH_RF95.h>
213 /// \brief Driver to send and receive unaddressed, unreliable datagrams via a LoRa
214 /// capable radio transceiver.
215 ///
216 /// For Semtech SX1276/77/78 and HopeRF RFM95/96/97/98 and other similar LoRa capable radios.
217 /// Based on http://www.hoperf.com/upload/rf/RFM95_96_97_98W.pdf
218 /// and http://www.hoperf.cn/upload/rfchip/RF96_97_98.pdf
219 /// and http://www.semtech.com/images/datasheet/LoraDesignGuide_STD.pdf
220 /// and http://www.semtech.com/images/datasheet/sx1276.pdf
221 /// FSK/GFSK/OOK modes are not (yet) supported.
222 ///
223 /// Works with
224 /// - the excellent MiniWirelessLoRa from Anarduino http://www.anarduino.com/miniwireless
225 ///
226 /// \par Overview
227 ///
228 /// This class provides basic functions for sending and receiving unaddressed,
229 /// unreliable datagrams of arbitrary length to 251 octets per packet.
230 ///
231 /// Manager classes may use this class to implement reliable, addressed datagrams and streams,
232 /// mesh routers, repeaters, translators etc.
233 ///
234 /// Naturally, for any 2 radios to communicate that must be configured to use the same frequency and
235 /// modulation scheme.
236 ///
237 /// This Driver provides an object-oriented interface for sending and receiving data messages with Hope-RF
238 /// RFM95/96/97/98(W) and compatible radio modules in LoRa mode.
239 ///
240 /// The Hope-RF (http://www.hoperf.com) RFM95/96/97/98(W) is a low-cost ISM transceiver
241 /// chip. It supports FSK, GFSK, OOK over a wide range of frequencies and
242 /// programmable data rates, and it also supports the proprietary LoRA (Long Range) mode, which
243 /// is the only mode supported in this RadioHead driver.
244 ///
245 /// This Driver provides functions for sending and receiving messages of up
246 /// to 251 octets on any frequency supported by the radio, in a range of
247 /// predefined Bandwidths, Spreading Factors and Coding Rates. Frequency can be set with
248 /// 61Hz precision to any frequency from 240.0MHz to 960.0MHz. Caution: most modules only support a more limited
249 /// range of frequencies due to antenna tuning.
250 ///
251 /// Up to 2 RFM95/96/97/98(W) modules can be connected to an Arduino (3 on a Mega),
252 /// permitting the construction of translators and frequency changers, etc.
253 ///
254 /// Support for other features such as transmitter power control etc is
255 /// also provided.
256 ///
257 /// Tested on MinWirelessLoRa with arduino-1.0.5
258 /// on OpenSuSE 13.1
259 ///
260 /// \par Packet Format
261 ///
262 /// All messages sent and received by this RH_RF95 Driver conform to this packet format:
263 ///
264 /// - LoRa mode:
265 /// - 8 symbol PREAMBLE
266 /// - Explicit header with header CRC (handled internally by the radio)
267 /// - 4 octets HEADER: (TO, FROM, ID, FLAGS)
268 /// - 0 to 251 octets DATA
269 /// - CRC (handled internally by the radio)
270 ///
271 /// \par Connecting RFM95/96/97/98 to Arduino
272 ///
273 /// We tested with Anarduino MiniWirelessLoRA, which is an Arduino Duemilanove compatible with a RFM96W
274 /// module on-board. Therefore it needs no connections other than the USB
275 /// programming connection and an antenna to make it work.
276 ///
277 /// If you have a bare RFM95/96/97/98 that you want to connect to an Arduino, you
278 /// might use these connections (untested): CAUTION: you must use a 3.3V type
279 /// Arduino, otherwise you will also need voltage level shifters between the
280 /// Arduino and the RFM95. CAUTION, you must also ensure you connect an
281 /// antenna.
282 ///
283 /// \code
284 /// Arduino RFM95/96/97/98
285 /// GND----------GND (ground in)
286 /// 3V3----------3.3V (3.3V in)
287 /// interrupt 0 pin D2-----------DIO0 (interrupt request out)
288 /// SS pin D10----------NSS (chip select in)
289 /// SCK pin D13----------SCK (SPI clock in)
290 /// MOSI pin D11----------MOSI (SPI Data in)
291 /// MISO pin D12----------MISO (SPI Data out)
292 /// \endcode
293 ///
294 /// With these connections, you can then use the default constructor RH_RF95().
295 /// You can override the default settings for the SS pin and the interrupt in
296 /// the RH_RF95 constructor if you wish to connect the slave select SS to other
297 /// than the normal one for your Arduino (D10 for Diecimila, Uno etc and D53
298 /// for Mega) or the interrupt request to other than pin D2 (Caution,
299 /// different processors have different constraints as to the pins available
300 /// for interrupts).
301 ///
302 /// It is possible to have 2 or more radios connected to one Arduino, provided
303 /// each radio has its own SS and interrupt line (SCK, SDI and SDO are common
304 /// to all radios)
305 ///
306 /// Caution: on some Arduinos such as the Mega 2560, if you set the slave
307 /// select pin to be other than the usual SS pin (D53 on Mega 2560), you may
308 /// need to set the usual SS pin to be an output to force the Arduino into SPI
309 /// master mode.
310 ///
311 /// Caution: Power supply requirements of the RFM module may be relevant in some circumstances:
312 /// RFM95/96/97/98 modules are capable of pulling 120mA+ at full power, where Arduino's 3.3V line can
313 /// give 50mA. You may need to make provision for alternate power supply for
314 /// the RFM module, especially if you wish to use full transmit power, and/or you have
315 /// other shields demanding power. Inadequate power for the RFM is likely to cause symptoms such as:
316 /// - reset's/bootups terminate with "init failed" messages
317 /// - random termination of communication after 5-30 packets sent/received
318 /// - "fake ok" state, where initialization passes fluently, but communication doesn't happen
319 /// - shields hang Arduino boards, especially during the flashing
320 ///
321 /// \par Interrupts
322 ///
323 /// The RH_RF95 driver uses interrupts to react to events in the RFM module,
324 /// such as the reception of a new packet, or the completion of transmission
325 /// of a packet. The RH_RF95 driver interrupt service routine reads status from
326 /// and writes data to the the RFM module via the SPI interface. It is very
327 /// important therefore, that if you are using the RH_RF95 driver with another
328 /// SPI based deviced, that you disable interrupts while you transfer data to
329 /// and from that other device. Use cli() to disable interrupts and sei() to
330 /// reenable them.
331 ///
332 /// \par Memory
333 ///
334 /// The RH_RF95 driver requires non-trivial amounts of memory. The sample
335 /// programs all compile to about 8kbytes each, which will fit in the
336 /// flash proram memory of most Arduinos. However, the RAM requirements are
337 /// more critical. Therefore, you should be vary sparing with RAM use in
338 /// programs that use the RH_RF95 driver.
339 ///
340 /// It is often hard to accurately identify when you are hitting RAM limits on Arduino.
341 /// The symptoms can include:
342 /// - Mysterious crashes and restarts
343 /// - Changes in behaviour when seemingly unrelated changes are made (such as adding print() statements)
344 /// - Hanging
345 /// - Output from Serial.print() not appearing
346 ///
347 /// \par Range
348 ///
349 /// We have made some simple range tests under the following conditions:
350 /// - rf95_client base station connected to a VHF discone antenna at 8m height above ground
351 /// - rf95_server mobile connected to 17.3cm 1/4 wavelength antenna at 1m height, no ground plane.
352 /// - Both configured for 13dBm, 434MHz, Bw = 125 kHz, Cr = 4/8, Sf = 4096chips/symbol, CRC on. Slow+long range
353 /// - Minimum reported RSSI seen for successful comms was about -91
354 /// - Range over flat ground through heavy trees and vegetation approx 2km.
355 /// - At 20dBm (100mW) otherwise identical conditions approx 3km.
356 /// - At 20dBm, along salt water flat sandy beach, 3.2km.
357 ///
358 /// It should be noted that at this data rate, a 12 octet message takes 2 seconds to transmit.
359 ///
360 /// At 20dBm (100mW) with Bw = 125 kHz, Cr = 4/5, Sf = 128chips/symbol, CRC on.
361 /// (Default medium range) in the conditions described above.
362 /// - Range over flat ground through heavy trees and vegetation approx 2km.
363 ///
364 /// \par Transmitter Power
365 ///
366 /// You can control the transmitter power on the RF transceiver
367 /// with the RH_RF95::setTxPower() function. The argument can be any of
368 /// +5 to +23
369 /// The default is 13. Eg:
370 /// \code
371 /// driver.setTxPower(10);
372 /// \endcode
373 ///
374 /// We have made some actual power measurements against
375 /// programmed power for Anarduino MiniWirelessLoRa (which has RFM96W-433Mhz installed)
376 /// - MiniWirelessLoRa RFM96W-433Mhz, USB power
377 /// - 30cm RG316 soldered direct to RFM96W module ANT and GND
378 /// - SMA connector
379 /// - 12db attenuator
380 /// - SMA connector
381 /// - MiniKits AD8307 HF/VHF Power Head (calibrated against Rohde&Schwartz 806.2020 test set)
382 /// - Tektronix TDS220 scope to measure the Vout from power head
383 /// \code
384 /// Program power Measured Power
385 /// dBm dBm
386 /// 5 5
387 /// 7 7
388 /// 9 8
389 /// 11 11
390 /// 13 13
391 /// 15 15
392 /// 17 16
393 /// 19 18
394 /// 20 20
395 /// 21 21
396 /// 22 22
397 /// 23 23
398 /// \endcode
399 /// (Caution: we dont claim laboratory accuracy for these measurements)
400 /// You would not expect to get anywhere near these powers to air with a simple 1/4 wavelength wire antenna.
