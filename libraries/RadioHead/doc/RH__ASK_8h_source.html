50 /// Outgoing message bits grouped as 6-bit words
51 /// 36 alternating 1/0 bits, followed by 12 bits of start symbol (together called the preamble)
52 /// Followed immediately by the 4-6 bit encoded byte count,
53 /// message buffer and 2 byte FCS
54 /// Each byte from the byte count on is translated into 2x6-bit words
55 /// Caution, each symbol is transmitted LSBit first,
56 /// but each byte is transmitted high nybble first

61 /// \class RH_ASK RH_ASK.h <RH_ASK.h>
62 /// \brief Driver to send and receive unaddressed, unreliable datagrams via inexpensive ASK (Amplitude Shift Keying) or
63 /// OOK (On Off Keying) RF transceivers.
64 ///
65 /// The message format and software technology is based on our VirtualWire library
66 /// (http://www.airspayce.com/mikem/arduino/VirtualWire), with which it is compatible.
67 /// See http://www.airspayce.com/mikem/arduino/VirtualWire.pdf for more details.
68 ///
69 /// RH_ASK is a Driver for Arduino, Maple and others that provides features to send short
70 /// messages, without addressing, retransmit or acknowledgment, a bit like UDP
71 /// over wireless, using ASK (amplitude shift keying). Supports a number of
72 /// inexpensive radio transmitters and receivers. All that is required is
73 /// transmit data, receive data and (for transmitters, optionally) a PTT
74 /// transmitter enable. Can also be used over various analog connections (not just a data radio),
75 /// such as the audio channel of an A/V sender, or long TTL lines.
76 ///
77 /// It is intended to be compatible with the RF Monolithics (www.rfm.com)
78 /// Virtual Wire protocol, but this has not been tested.
79 ///
80 /// Does not use the Arduino UART. Messages are sent with a training preamble,
81 /// message length and checksum. Messages are sent with 4-to-6 bit encoding
82 /// for good DC balance, and a CRC checksum for message integrity.
83 ///
84 /// But why not just use a UART connected directly to the
85 /// transmitter/receiver? As discussed in the RFM documentation, ASK receivers
86 /// require a burst of training pulses to synchronize the transmitter and
87 /// receiver, and also requires good balance between 0s and 1s in the message
88 /// stream in order to maintain the DC balance of the message. UARTs do not
89 /// provide these. They work a bit with ASK wireless, but not as well as this
90 /// code.
91 ///
92 /// \par Theory of operation
93 ///
94 /// See ASH Transceiver Software Designer's Guide of 2002.08.07
95 /// http://www.rfm.com/products/apnotes/tr_swg05.pdf
96 ///
97 /// \par Supported Hardware
98 ///
99 /// A range of communications
100 /// hardware is supported. The ones listed below are available in common retail
101 /// outlets in Australia and other countries for under $10 per unit. Many
102 /// other modules may also work with this software.
103 ///
104 /// Runs on a wide range of Arduino processors using Arduino IDE 1.0 or later.
105 /// Also runs on on Energia
106 /// with MSP430G2553 / G2452 and Arduino with ATMega328 (courtesy Yannick DEVOS - XV4Y),
107 /// but untested by us. It also runs on Teensy 3.0 (courtesy of Paul
108 /// Stoffregen), but untested by us. Also compiles and runs on ATtiny85 in
109 /// Arduino environment, courtesy r4z0r7o3. Also compiles on maple-ide-v0.0.12,
110 /// and runs on Maple, flymaple 1.1 etc. Runs on ATmega8/168 (Arduino Diecimila,
111 /// Uno etc), ATmega328 and can run on almost any other AVR8 platform,
112 /// without relying on the Arduino framework, by properly configuring the
113 /// library editing the RH_ASK.h header file for describing the access
114 /// to IO pins and for setting up the timer.
115 ///
116 /// - Receivers
117 /// - RX-B1 (433.92MHz) (also known as ST-RX04-ASK)
118 /// - RFM83C from HopeRF http://www.hoperfusa.com/details.jsp?pid=126
119 /// - Transmitters:
120 /// - TX-C1 (433.92MHz)
121 /// - RFM85 from HopeRF http://www.hoperfusa.com/details.jsp?pid=127
122 /// - Transceivers
123 /// - DR3100 (433.92MHz)
124 ///
125 /// \par Connecting to Arduino
126 ///
127 /// Most transmitters can be connected to Arduino like this:
128 
129 /// \code
130 /// Arduino Transmitter
131 /// GND------------------------------GND
132 /// D12------------------------------Data
133 /// 5V-------------------------------VCC
134 /// \endcode
135 ///
136 /// Most receivers can be connected to Arduino like this:
137 /// \code
138 /// Arduino Receiver
139 /// GND------------------------------GND
140 /// D11------------------------------Data
141 /// 5V-------------------------------VCC
142 /// SHUT (not connected)
143 /// WAKEB (not connected)
144 /// GND |
145 /// ANT |- connect to your antenna syetem
146 /// \endcode
147 ///
148 /// RH_ASK works with ATTiny85, using Arduino 1.0.5 and tinycore from
149 /// https://code.google.com/p/arduino-tiny/downloads/detail?name=arduino-tiny-0100-0018.zip
150 /// Tested with the examples ask_transmitter and ask_receiver on ATTiny85.
151 /// Caution: The RAM memory requirements on an ATTiny85 are *very* tight. Even the bare bones
152 /// ask_transmitter sketch barely fits in eh RAM available on the ATTiny85. Its unlikely to work on
153 /// smaller ATTinys such as the ATTiny45 etc. If you have wierd behaviour, consider
154 /// reducing the size of RH_ASK_MAX_PAYLOAD_LEN to the minimum you can work with.
155 /// Caution: the default internal clock speed on an ATTiny85 is 1MHz. You MUST set the internal clock speed
156 /// to 8MHz. You can do this with Arduino IDE, tineycore and ArduinoISP by setting the board type to "ATtiny85@8MHz',
157 /// setting theProgrammer to 'Arduino as ISP' and selecting Tools->Burn Bootloader. This does not actually burn a
158 /// bootloader into the tiny, it just changes the fuses so the chip runs at 8MHz.
159 /// If you run the chip at 1MHz, you will get RK_ASK speeds 1/8th of the expected.
160 ///
161 /// Initialise RH_ASK for ATTiny85 like this:
162 /// // #include <SPI.h> // comment this out, not needed
163 /// RH_ASK driver(2000, 4, 3); // 200bps, TX on D3 (pin 2), RX on D4 (pin 3)
164 /// then:
165 /// Connect D3 (pin 2) as the output to the transmitter
166 /// Connect D4 (pin 3) as the input from the receiver.
167 ///
168 ///
169 /// For testing purposes you can connect 2 Arduino RH_ASK instances directly, by
170 /// connecting pin 12 of one to 11 of the other and vice versa, like this for a duplex connection:
171 ///
172 /// \code
173 /// Arduino 1 wires Arduino 1
174 /// D11-----------------------------D12
175 /// D12-----------------------------D11
176 /// GND-----------------------------GND
177 /// \endcode
178 ///
179 /// You can also connect 2 RH_ASK instances over a suitable analog
180 /// transmitter/receiver, such as the audio channel of an A/V transmitter/receiver. You may need
181 /// buffers at each end of the connection to convert the 0-5V digital output to a suitable analog voltage.
182 ///
183 /// Measured power output from RFM85 at 5V was 18dBm.
184 ///
185 /// \par Timers
186 /// The RH_ASK driver uses a timer-driven interrupt to generate 8 interrupts per bit period. RH_ASK
187 /// takes over a timer on Arduino-like platforms. By default it takes over Timer 1. You can force it
188 /// to use Timer 2 instead by enabling the define RH_ASK_ARDUINO_USE_TIMER2 near the top of RH_ASK.cpp
189 ///
190 /// Caution: ATTiny85 has only 2 timers, one (timer 0) usually used for
191 /// millis() and one (timer 1) for PWM analog outputs. The RH_ASK Driver
192 /// library, when built for ATTiny85, takes over timer 0, which prevents use
193 /// of millis() etc but does permit analog outputs. This will affect the accuracy of millis() and time
194 /// measurement.
