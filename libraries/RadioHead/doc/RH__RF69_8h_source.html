292 /// \class RH_RF69 RH_RF69.h <RH_RF69.h>
293 /// \brief Driver to send and receive unaddressed, unreliable datagrams via an RF69 and compatible radio transceiver.
294 ///
295 /// Works with
296 /// - the excellent Moteino and Moteino-USB
297 /// boards from LowPowerLab http://lowpowerlab.com/moteino/
298 /// - compatible chips and modules such as RFM69W, RFM69HW, RFM69CW, RFM69HCW (Semtech SX1231, SX1231H),
299 /// - RFM69 modules from http://www.hoperfusa.com such as http://www.hoperfusa.com/details.jsp?pid=145
300 /// - Anarduino MiniWireless -CW and -HW boards http://www.anarduino.com/miniwireless/ including
301 /// the marvellous high powered MinWireless-HW (with 20dBm output for excellent range)
302 ///
303 /// \par Overview
304 ///
305 /// This class provides basic functions for sending and receiving unaddressed,
306 /// unreliable datagrams of arbitrary length to 64 octets per packet.
307 ///
308 /// Manager classes may use this class to implement reliable, addressed datagrams and streams,
309 /// mesh routers, repeaters, translators etc.
310 ///
311 /// Naturally, for any 2 radios to communicate that must be configured to use the same frequency and
312 /// modulation scheme.
313 ///
314 /// This Driver provides an object-oriented interface for sending and receiving data messages with Hope-RF
315 /// RF69B and compatible radio modules, such as the RFM69 module.
316 ///
317 /// The Hope-RF (http://www.hoperf.com) RF69 is a low-cost ISM transceiver
318 /// chip. It supports FSK, GFSK, OOK over a wide range of frequencies and
319 /// programmable data rates. It also suports AES encryption of up to 64 octets
320 /// of payload It is available prepackaged on modules such as the RFM69W. And
321 /// such modules can be prepacked on processor boards such as the Moteino from
322 /// LowPowerLabs (which is what we used to develop the RH_RF69 driver)
323 ///
324 /// This Driver provides functions for sending and receiving messages of up
325 /// to 60 octets on any frequency supported by the RF69, in a range of
326 /// predefined data rates and frequency deviations. Frequency can be set with
327 /// 61Hz precision to any frequency from 240.0MHz to 960.0MHz. Caution: most modules only support a more limited
328 /// range of frequencies due to antenna tuning.
329 ///
330 /// Up to 2 RF69B modules can be connected to an Arduino (3 on a Mega),
331 /// permitting the construction of translators and frequency changers, etc.
332 ///
333 /// The following modulation types are suppported with a range of modem configurations for
334 /// common data rates and frequency deviations:
335 /// - GFSK Gaussian Frequency Shift Keying
336 /// - FSK Frequency Shift Keying
337 ///
338 /// Support for other RF69 features such as on-chip temperature measurement,
339 /// transmitter power control etc is also provided.
340 ///
341 /// Tested on USB-Moteino with arduino-1.0.5
342 /// on OpenSuSE 13.1
343 ///
344 /// \par Packet Format
345 ///
346 /// All messages sent and received by this RH_RF69 Driver conform to this packet format:
347 ///
348 /// - 4 octets PREAMBLE
349 /// - 2 octets SYNC 0x2d, 0xd4 (configurable, so you can use this as a network filter)
350 /// - 1 octet RH_RF69 payload length
351 /// - 4 octets HEADER: (TO, FROM, ID, FLAGS)
352 /// - 0 to 60 octets DATA
353 /// - 2 octets CRC computed with CRC16(IBM), computed on HEADER and DATA
354 ///
355 /// For technical reasons, the message format is not protocol compatible with the
356 /// 'HopeRF Radio Transceiver Message Library for Arduino'
357 /// http://www.airspayce.com/mikem/arduino/HopeRF from the same author. Nor is
358 /// it compatible with messages sent by 'Virtual Wire'
359 /// http://www.airspayce.com/mikem/arduino/VirtualWire.pdf also from the same
360 /// author. Nor is it compatible with messages sent by 'RF22'
361 /// http://www.airspayce.com/mikem/arduino/RF22 also from the same author.
362 ///
363 /// \par Connecting RFM-69 to Arduino
364 ///
365 /// We tested with Moteino, which is an Arduino Uno compatible with the RFM69W
366 /// module on-board. Therefore it needs no connections other than the USB
367 /// programming connection and an antenna to make it work.
368 ///
369 /// If you have a bare RFM69W that you want to connect to an Arduino, you
370 /// might use these connections (untested): CAUTION: you must use a 3.3V type
371 /// Arduino, otherwise you will also need voltage level shifters between the
372 /// Arduino and the RFM69. CAUTION, you must also ensure you connect an
373 /// antenna
374 ///
375 /// \code
376 /// Arduino RFM69W
377 /// GND----------GND (ground in)
378 /// 3V3----------3.3V (3.3V in)
379 /// interrupt 0 pin D2-----------DIO0 (interrupt request out)
380 /// SS pin D10----------NSS (chip select in)
381 /// SCK pin D13----------SCK (SPI clock in)
382 /// MOSI pin D11----------MOSI (SPI Data in)
383 /// MISO pin D12----------MISO (SPI Data out)
384 /// \endcode
385 ///
386 /// With these connections, you can then use the default constructor RH_RF69().
387 /// You can override the default settings for the SS pin and the interrupt in
388 /// the RH_RF69 constructor if you wish to connect the slave select SS to other
389 /// than the normal one for your Arduino (D10 for Diecimila, Uno etc and D53
390 /// for Mega) or the interrupt request to other than pin D2 (Caution,
391 /// different processors have different constraints as to the pins available
392 /// for interrupts).
393 ///
394 /// If you have a Teensy 3.1 and a compatible RFM69 breakout board, you will need to
395 /// construct the RH_RF69 instance like this:
396 /// \code
397 /// RH_RF69 driver(15, 16);
398 /// \endcode
399 ///
400 /// If you have a MoteinoMEGA https://lowpowerlab.com/shop/moteinomega
401 /// with RFM69 on board, you dont need to make any wiring connections
402 /// (the RFM69 module is soldered onto the MotienoMEGA), but you must initialise the RH_RF69
403 /// constructor like this:
404 /// \code
405 /// RH_RF69 driver(4, 2);
406 /// \endcode
407 /// Make sure you have the MoteinoMEGA core installed in your Arduino hardware folder as described in the
408 /// documentation for the MoteinoMEGA.
409 ///
410 /// It is possible to have 2 or more radios connected to one Arduino, provided
411 /// each radio has its own SS and interrupt line (SCK, SDI and SDO are common
412 /// to all radios)
413 ///
414 /// Caution: on some Arduinos such as the Mega 2560, if you set the slave
415 /// select pin to be other than the usual SS pin (D53 on Mega 2560), you may
416 /// need to set the usual SS pin to be an output to force the Arduino into SPI
417 /// master mode.
418 ///
419 /// Caution: Power supply requirements of the RF69 module may be relevant in some circumstances:
420 /// RF69 modules are capable of pulling 45mA+ at full power, where Arduino's 3.3V line can
421 /// give 50mA. You may need to make provision for alternate power supply for
422 /// the RF69, especially if you wish to use full transmit power, and/or you have
423 /// other shields demanding power. Inadequate power for the RF69 is likely to cause symptoms such as:
424 /// -reset's/bootups terminate with "init failed" messages
425 /// -random termination of communication after 5-30 packets sent/received
426 /// -"fake ok" state, where initialization passes fluently, but communication doesn't happen
427 /// -shields hang Arduino boards, especially during the flashing
428 /// \par Interrupts
429 ///
430 /// The RH_RF69 driver uses interrupts to react to events in the RF69 module,
431 /// such as the reception of a new packet, or the completion of transmission
432 /// of a packet. The RH_RF69 driver interrupt service routine reads status from
433 /// and writes data to the the RF69 module via the SPI interface. It is very
434 /// important therefore, that if you are using the RH_RF69 driver with another
435 /// SPI based deviced, that you disable interrupts while you transfer data to
436 /// and from that other device. Use cli() to disable interrupts and sei() to
437 /// reenable them.
438 ///
439 /// \par Memory
440 ///
441 /// The RH_RF69 driver requires non-trivial amounts of memory. The sample
442 /// programs above all compile to about 8kbytes each, which will fit in the
443 /// flash proram memory of most Arduinos. However, the RAM requirements are
444 /// more critical. Therefore, you should be vary sparing with RAM use in
445 /// programs that use the RH_RF69 driver.
446 ///
447 /// It is often hard to accurately identify when you are hitting RAM limits on Arduino.
448 /// The symptoms can include:
449 /// - Mysterious crashes and restarts
450 /// - Changes in behaviour when seemingly unrelated changes are made (such as adding print() statements)
451 /// - Hanging
452 /// - Output from Serial.print() not appearing
453 ///
454 /// \par Automatic Frequency Control (AFC)
455 ///
456 /// The RF69 module is configured by the RH_RF69 driver to always use AFC.
457 ///
458 /// \par Transmitter Power
