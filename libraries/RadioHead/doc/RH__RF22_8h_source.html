403 /// - Silicon Labs Si4430/31/32 based modules. S4432 is equivalent to RF22. Si4431/30 is equivalent to RF23.
404 ///
405 /// Data based on https://www.sparkfun.com/datasheets/Wireless/General/RFM22B.pdf
406 ///
407 /// \par Overview
408 ///
409 /// This base class provides basic functions for sending and receiving unaddressed,
410 /// unreliable datagrams of arbitrary length to 255 octets per packet.
411 ///
412 /// Manager classes may use this class to implement reliable, addressed datagrams and streams,
413 /// mesh routers, repeaters, translators etc.
414 ///
415 /// On transmission, the TO and FROM addresses default to 0x00, unless changed by a subclass.
416 /// On reception the TO addressed is checked against the node address (defaults to 0x00) or the
417 /// broadcast address (which is 0xff). The ID and FLAGS are set to 0, and not checked by this class.
418 /// This permits use of the this base RH_RF22 class as an
419 /// unaddressed, unreliable datagram service without the use of one the RadioHead Manager classes.
420 ///
421 /// Naturally, for any 2 radios to communicate that must be configured to use the same frequency and
422 /// modulation scheme.
423 ///
424 /// \par Details
425 ///
426 /// This Driver provides an object-oriented interface for sending and receiving data messages with Hope-RF
427 /// RF22 and RF23 based radio modules, and compatible chips and modules,
428 /// including the RFM22B transceiver module such as
429 /// this bare module: http://www.sparkfun.com/products/10153
430 /// and this shield: http://www.sparkfun.com/products/11018
431 /// and this module: http://www.hoperfusa.com/details.jsp?pid=131
432 /// and this integrated board: http://www.anarduino.com/miniwireless
433 /// and RF23BP modules such as this http://www.anarduino.com/details.jsp?pid=130
434 ///
435 /// The Hope-RF (http://www.hoperf.com) RFM22B (http://www.hoperf.com/rf_fsk/fsk/RFM22B.htm)
436 /// is a low-cost ISM transceiver module. It supports FSK, GFSK, OOK over a wide
437 /// range of frequencies and programmable data rates.
438 /// Manual can be found at https://www.sparkfun.com/datasheets/Wireless/General/RFM22.PDF
439 ///
440 /// This library provides functions for sending and receiving messages of up to 255 octets on any
441 /// frequency supported by the RF22B, in a range of predefined data rates and frequency deviations.
442 /// Frequency can be set with 312Hz precision to any frequency from 240.0MHz to 960.0MHz.
443 ///
444 /// Up to 3 RF22B modules can be connected to an Arduino, permitting the construction of translators
445 /// and frequency changers, etc.
446 ///
447 /// The following modulation types are suppported with a range of modem configurations for
448 /// common data rates and frequency deviations:
449 /// - GFSK Gaussian Frequency Shift Keying
450 /// - FSK Frequency Shift Keying
451 /// - OOK On-Off Keying
452 ///
453 /// Support for other RF22B features such as on-chip temperature measurement, analog-digital
454 /// converter, transmitter power control etc is also provided.
455 ///
456 /// Tested on Arduino Diecimila, Uno and Mega with arduino-0021, 1.0.5
457 /// on OpenSuSE 13.1 and avr-libc-1.6.1-1.15,
458 /// cross-avr-binutils-2.19-9.1, cross-avr-gcc-4.1.3_20080612-26.5.
459 /// With HopeRF RFM22 modules that appear to have RF22B chips on board:
460 /// - Device Type Code = 0x08 (RX/TRX)
461 /// - Version Code = 0x06
462 /// Works on Duo. Works with Sparkfun RFM22 Wireless shields. Works with RFM22 modules from http://www.hoperfusa.com/
463 /// Works with Arduino 1.0 to at least 1.0.5. Works on Maple, Flymaple, Uno32.
464 ///
465 /// \par Packet Format
466 ///
467 /// All messages sent and received by this Driver must conform to this packet format:
468 ///
469 /// - 8 nibbles (4 octets) PREAMBLE
470 /// - 2 octets SYNC 0x2d, 0xd4
471 /// - 4 octets HEADER: (TO, FROM, ID, FLAGS)
472 /// - 1 octet LENGTH (0 to 255), number of octets in DATA
473 /// - 0 to 255 octets DATA
474 /// - 2 octets CRC computed with CRC16(IBM), computed on HEADER, LENGTH and DATA
475 ///
476 /// For technical reasons, the message format is not protocol compatible with the
477 /// 'HopeRF Radio Transceiver Message Library for Arduino' http://www.airspayce.com/mikem/arduino/HopeRF from the same author. Nor is it compatible with
478 /// 'Virtual Wire' http://www.airspayce.com/mikem/arduino/VirtualWire.pdf also from the same author.
479 ///
480 /// \par Connecting RFM-22 to Arduino

482 /// If you have the Sparkfun RFM22 Shield (https://www.sparkfun.com/products/11018)

484 /// just add headers and plug it in to an Arduino (but not and Arduino Mega, see below)
485 ///

487 /// the 3 x SPI pins (SCK, SDI, SDO), a Slave Select pin and an interrupt pin.
488 ///

490 /// RX_ANT pins of the RFM22 in order to control the antenna connection properly. The RH_RF22

492 /// control TX_ANT and RX_ANT input pins respectively automatically. On RFM22,

627 /// \par Interrupts
628 ///
629 /// The Driver uses interrupts to react to events in the RF22 module,
630 /// such as the reception of a new packet, or the completion of transmission of a packet.
631 /// The RH_RF22 interrupt service routine reads status from and writes data
632 /// to the the RF22 module via the SPI interface. It is very important therefore,
633 /// that if you are using the RF22 library with another SPI based deviced, that you
634 /// disable interrupts while you transfer data to and from that other device.
635 /// Use cli() to disable interrupts and sei() to reenable them.
636 ///
637 /// \par SPI Interface
638 ///
639 /// The RF22 module uses the SPI bus to communicate with the Arduino. Arduino
640 /// IDE includes a hardware SPI class to communicate with SPI devices using
641 /// the SPI facilities built into the Atmel chips, over the standard designated
642 /// SPI pins MOSI, MISO, SCK, which are usually on Arduino pins 11, 12 and 13
643 /// respectively (or 51, 50, 52 on a Mega).
644 ///
645 /// By default, the RH_RF22 Driver uses the Hardware SPI interface to
646 /// communicate with the RF22 module. However, if your RF22 SPI is connected to
647 /// the Arduino through non-standard pins, or the standard Hardware SPI
648 /// interface will not work for you, you can instead use a bit-banged Software
649 /// SPI class RHSoftwareSPI, which can be configured to work on any Arduino digital IO pins.
650 /// See the documentation of RHSoftwareSPI for details.
651 ///
652 /// The advantages of the Software SPI interface are that it can be used on
653 /// any Arduino pins, not just the usual dedicated hardware pins. The
654 /// disadvantage is that it is significantly slower then hardware.
655 /// If you observe reliable behaviour with the default hardware SPI RHHardwareSPI, but unreliable behaviour
656 /// with Software SPI RHSoftwareSPI, it may be due to slow CPU performance.
657 ///
658 /// Initialisation example with hardware SPI
659 /// \code
660 /// #include <RH_RF22.h>
661 /// RH_RF22 driver;
662 /// RHReliableDatagram manager(driver, CLIENT_ADDRESS);
663 /// \endcode
664 ///
665 /// Initialisation example with software SPI
666 /// \code
667 /// #include <RH_RF22.h>
668 /// #include <RHSoftwareSPI.h>
669 /// RHSoftwareSPI spi;
670 /// RH_RF22 driver(10, 2, spi);
671 /// RHReliableDatagram manager(driver, CLIENT_ADDRESS);
672 /// \endcode

674 /// \par Memory
675 ///
676 /// The RH_RF22 Driver requires non-trivial amounts of memory. The sample programs all compile to
677 /// about 9 to 14kbytes each on Arduino, which will fit in the flash proram memory of most Arduinos. However,
678 /// the RAM requirements are more critical. Most sample programs above will run on Duemilanova,
679 /// but not on Diecimila. Even on Duemilanova, the RAM requirements are very close to the
680 /// available memory of 2kbytes. Therefore, you should be vary sparing with RAM use in programs that use
681 /// the RH_RF22 Driver on Duemilanova.
682 ///
683 /// The sample RHRouter and RHMesh programs compile to about 14kbytes,
684 /// and require more RAM than the others.
685 /// They will not run on Duemilanova or Diecimila, but will run on Arduino Mega.
686 ///
687 /// It is often hard to accurately identify when you are hitting RAM limits on Arduino.
688 /// The symptoms can include:
689 /// - Mysterious crashes and restarts
690 /// - Changes in behaviour when seemingly unrelated changes are made (such as adding print() statements)
691 /// - Hanging
692 /// - Output from Serial.print() not appearing
693 ///
694 /// With an Arduino Mega, with 8 kbytes of SRAM, there is much more RAM headroom for
695 /// your own elaborate programs.
696 /// This library is reported to work with Arduino Pro Mini, but that has not been tested by me.
697 ///
698 /// The RF22M modules use an inexpensive crystal to control the frequency synthesizer, and therfore you can expect
699 /// the transmitter and receiver frequencies to be subject to the usual inaccuracies of such crystals. The RF22
700 /// contains an AFC circuit to compensate for differences in transmitter and receiver frequencies.
701 /// It does this by altering the receiver frequency during reception by up to the pull-in frequency range.
702 /// This RF22 library enables the AFC and by default sets the pull-in frequency range to
703 /// 0.05MHz, which should be sufficient to handle most situations. However, if you observe unexplained packet losses
704 /// or failure to operate correctly all the time it may be because your modules have a wider frequency difference, and
705 /// you may need to set the afcPullInRange to a different value, using setFrequency();
706 ///
707 /// \par Transmitter Power
708 ///
709 /// You can control the transmitter power on the RF22 and RF23 transceivers
710 /// with the RH_RF22::setTxPower() function. The argument can be any of the
711 /// RH_RF22_TXPOW_* (for RFM22) or RH_RF22_RF23B_TXPOW_* (for RFM23) values.
712 /// The default is RH_RF22_TXPOW_8DBM/RH_RF22_RF23B_TXPOW_1DBM . Eg:
713 /// \code
714 /// driver.setTxPower(RH_RF22_TXPOW_2DBM);
715 /// \endcode
716 ///
717 /// The RF23BP has higher power capability, there are
718 /// several power settings that are specific to the RF23BP only:
719 ///
720 /// - RH_RF22_RF23BP_TXPOW_28DBM
721 /// - RH_RF22_RF23BP_TXPOW_29DBM
722 /// - RH_RF22_RF23BP_TXPOW_38DBM
723 ///
724 /// CAUTION: the high power settings available on the RFM23BP require
725 /// significant power supply current. For example at +30dBm, the typical chip
726 /// supply current is 550mA. This will overwhelm some small CPU board power
727 /// regulators and USB supplies. If you use this chip at high power make sure
728 /// you have an adequate supply current providing full 5V to the RFM23BP (and
729 /// the CPU if required), otherwise you can expect strange behaviour like
730 /// hanging, stopping, incorrect power levels, RF power amp overheating etc.
731 /// You must also ensure that the RFM23BP GPIO pins are connected to the
732 /// antenna switch control pins like so:
733 ////
734 /// \code
735 /// GPIO0 <-> RXON
736 /// GPIO1 <-> TXON
737 /// \endcode
738 ///
739 /// The RF output impedance of the RFM22BP module is 50 ohms. In our
740 /// experiments we found that the most critical issue (besides a suitable
741 /// power supply) is to ensure that the antenna impedance is also near 50
742 /// ohms. Connecting a simple 1/4 wavelength (ie a 17.3cm single wire)
743 /// directly to the antenna output <b>will not work at full 30dBm power</b>,
744 /// and will result in the transmitter hanging and/or the power amp
745 /// overheating. Connect a proper 50 ohm impedance transmission line or
746 /// antenna, and prevent RF radiation into the radio and arduino modules,
747 /// in order to get full, reliable power. Our tests show that a 433MHz
748 /// RFM23BP feeding a 50 ohm transmission line with a VHF discone antenna at
749 /// the end results in full power output and the power amp transistor on the
750 /// RFM22BP module runnning slightly warm but not hot. We recommend you use
751 /// the services of a competent RF engineer when trying to use this high power
752 /// module.
753 ///
754 /// Note: with RFM23BP, the reported maximum possible power when operating on 3.3V is 27dBm.
755 ///
756 /// We have made some actual power measurements against
757 /// programmed power for Sparkfun RFM22 wireless module under the following conditions:
758 /// - Sparkfun RFM22 wireless module, Duemilanove, USB power
759 /// - 10cm RG58C/U soldered direct to RFM22 module ANT and GND
760 /// - bnc connecteor
761 /// - 12dB attenuator
762 /// - BNC-SMA adapter
763 /// - MiniKits AD8307 HF/VHF Power Head (calibrated against Rohde&Schwartz 806.2020 test set)
764 /// - Tektronix TDS220 scope to measure the Vout from power head
765 /// \code
766 /// Program power Measured Power
767 /// dBm dBm
768 /// 1 -5.6
769 /// 2 -3.8
770 /// 5 -2.2
771 /// 8 -0.6
772 /// 11 1.2
773 /// 14 11.6
774 /// 17 14.4
775 /// 20 18.0
776 /// \endcode
777 /// (Caution: we dont claim laboratory accuracy for these measurements)
778 /// You would not expect to get anywhere near these powers to air with a simple 1/4 wavelength wire antenna.
779 ///
780 /// \par Performance
781 ///
782 /// Some simple speed performance tests have been conducted.
783 /// In general packet transmission rate will be limited by the modulation scheme.
784 /// Also, if your code does any slow operations like Serial printing it will also limit performance.
785 /// We disabled any printing in the tests below.
786 /// We tested with RH_RF22::GFSK_Rb125Fd125, which is probably the fastest scheme available.
787 /// We tested with a 13 octet message length, over a very short distance of 10cm.

789 /// Transmission (no reply) tests with modulation RH_RF22::GFSK_Rb125Fd125 and a
